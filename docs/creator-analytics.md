# Creator Analytics

## Goal

Give creators and operators a clear view of how AI-generated content is performing across three core analytics areas:

1. **Generation volume** — how many generations creators produce over time.
2. **Approval rate** — how often generated assets are approved for use or publication.
3. **Per-pack engagement** — how each content pack performs after approval and distribution.

## Primary dashboard cards

### Generation volume

Shows total generations created in the selected date range.

Recommended breakdowns:

- Daily generation count
- Unique creators generating content
- Unique packs receiving generated content
- Average generations per creator
- Average generations per pack

### Approval rate

Shows the share of generated assets that were approved.

Formula:

```text
approval_rate = approved_generations / reviewed_generations
```

Recommended states:

- `pending` — generated but not yet reviewed
- `approved` — accepted for use
- `rejected` — rejected during review

The denominator should use reviewed generations only, not all generations. This avoids making pending review queues look like failed approvals.

### Per-pack engagement

Shows engagement generated by approved content within each pack.

Recommended tracked events:

- `view`
- `open`
- `like`
- `share`
- `save`
- `click`
- `purchase`

Recommended metrics:

- Views
- Opens
- Likes
- Shares
- Saves
- Clicks
- Purchases
- Engagement score
- Click-through rate
- Conversion rate

## Event model

Analytics should be event-based so reporting can be recomputed consistently as the product evolves.

### Generation event

Emitted when a creator generates an asset.

Required fields:

- `id`
- `creator_id`
- `pack_id`
- `asset_id`
- `model`
- `created_at`

Optional fields:

- `prompt_template_id`
- `generation_type`
- `metadata`

### Review event

Emitted when an asset is approved or rejected.

Required fields:

- `id`
- `generation_id`
- `review_status`
- `reviewed_by`
- `reviewed_at`

Valid `review_status` values:

- `approved`
- `rejected`

### Engagement event

Emitted when an end user interacts with an approved asset or pack.

Required fields:

- `id`
- `creator_id`
- `pack_id`
- `asset_id`
- `event_type`
- `occurred_at`

Optional fields:

- `user_id`
- `session_id`
- `metadata`

## Reporting windows

The dashboard should support:

- Last 7 days
- Last 30 days
- Month to date
- Quarter to date
- Custom date range

All queries should be scoped by inclusive `start_date` and exclusive `end_date` to avoid double-counting across adjacent periods.

## Recommended dashboard layout

1. Top cards
   - Total generations
   - Reviewed generations
   - Approval rate
   - Total engagement events
2. Generation trend
   - Daily bar or line chart
3. Approval funnel
   - Pending, approved, rejected counts
4. Pack performance table
   - Pack name or ID
   - Generations
   - Approval rate
   - Views
   - Likes
   - Shares
   - Saves
   - Clicks
   - Purchases
   - Engagement score
5. Creator leaderboard
   - Creator
   - Generations
   - Approval rate
   - Best-performing pack

## Engagement score

Use a weighted score so high-intent events count more than passive impressions.

Default weights:

```text
view = 1
open = 2
like = 3
save = 4
share = 5
click = 5
purchase = 10
```

Formula:

```text
engagement_score = sum(event_count * event_weight)
```

Weights should be configurable because product goals may change.

## API contract

A creator analytics endpoint should accept:

```ts
type CreatorAnalyticsParams = {
  creatorId?: string;
  packId?: string;
  startDate: string;
  endDate: string;
};
```

And return:

```ts
type CreatorAnalyticsSummary = {
  generationVolume: number;
  reviewedGenerations: number;
  approvedGenerations: number;
  rejectedGenerations: number;
  approvalRate: number | null;
  engagementEvents: number;
  engagementScore: number;
  packs: PackAnalytics[];
};
```

## Implementation notes

- Store raw events append-only.
- Build dashboard queries from aggregated views or materialized views.
- Use `reviewed_generations` as the approval-rate denominator.
- Keep pending generations visible as queue health, but separate from approval-rate calculations.
- Use UTC timestamps at storage boundaries and convert display dates in the UI.
- Add indexes on `creator_id`, `pack_id`, and event timestamps.
- For large event volumes, roll up daily metrics into a separate aggregate table.

## Acceptance criteria

- Creators can see total generation volume over the selected date range.
- Creators can see approved, rejected, pending, and reviewed generation counts.
- Approval rate is calculated as approved divided by reviewed.
- Creators can compare engagement by pack.
- Pack analytics include at least views, likes, shares, saves, clicks, purchases, and weighted engagement score.
- Analytics can be filtered by creator, pack, and date range.
- Metrics do not double-count across adjacent date ranges.
